# 06 — Using I18N.t() in JavaScript

---

#### 1. The Problem
HTML data-i18n attributes handle static text. But what about text generated by JavaScript? Like:
- Alert messages
- Dynamic button text
- Console logs for debugging
- Error messages from API calls

#### 2. The Solution: I18N.t()
```js
const message = I18N.t('search.fetch_error')
alert(message)
```

#### 3. Interpolation (Dynamic Values)
```js
const count = 2
const text = I18N.t('search.only_seats_left', { count })
// Result: "Only 2 seats left!"
```

#### 4. Real Examples from the Codebase

**Booking page (booking.html inline JS):**
```js
// Before:
alert('Please select a seat first.')

// After:
alert(I18N.t('booking.select_seat_alert'))
```

**Agent scanner (scanner.js):**
```js
// Before:
statusEl.textContent = '✅ VALID TICKET'

// After:
statusEl.textContent = '✅ ' + I18N.t('agent.valid_ticket')
```

**Agent sell (sell.js):**
```js
// Before:
button.innerHTML = '✅ TICKET PRINTED'

// After:
button.innerHTML = '✅ ' + (window.I18N ? I18N.t('agent.ticket_printed') : 'TICKET PRINTED')
```
(The `window.I18N` check is a safety guard in case i18n hasn't loaded yet.)

#### 5. The onChange() Callback
When the user switches language, any dynamically-generated text needs to update too.

```js
I18N.onChange((lang) => {
  // Re-render dynamic UI elements in the new language
  updateBookingSummary()
  updateTimerDisplay()
})
```

#### 6. The Custom Event Alternative
```js
document.addEventListener('i18n:change', (e) => {
  console.log('Language changed to:', e.detail.lang)
  refreshUI()
})
```

#### 7. Safety Pattern
Always provide a fallback when using I18N.t() in case it's not loaded:
```js
function safeT(key, fallback) {
  return window.I18N ? I18N.t(key) : fallback
}
// Usage:
const text = safeT('agent.valid_ticket', 'VALID TICKET')
```

#### 8. When NOT to Use I18N.t()
- Data that comes from the server (passenger names, prices)
- Data that changes every page load (ticket references, dates)
- Pure CSS content (::before, ::after pseudo-elements)
