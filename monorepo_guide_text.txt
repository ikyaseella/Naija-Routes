Section 1: What Are We Building and How Does It All Connect?
Before we look at any code or folder structure, you need to understand the big picture — what the Naija Routes system is made of and how the pieces talk to each other. Think of it like the parts of a transport business.
💡
The simple mental model
Naija Routes is like a transport company headquarters. The APPS are the different offices (traveller front desk, agent desk, operator office, admin war room). The SERVER is the back office where all the real work happens. The DATABASE is the filing cabinet. They all share the same filing cabinet but have their own desks.

Section 2: What Is a Monorepo and Why Are We Using One?
This word — monorepo — sounds complicated but the idea is simple. Let us explain it with a story first.
2.1  The Problem Without a Monorepo
Imagine the Naija Routes team decides to keep each part of the project in a separate GitHub repository:
Repository 1: naija-routes-web (the traveller website)
Repository 2: naija-routes-agent (the agent app)
Repository 3: naija-routes-operator (the operator dashboard)
Repository 4: naija-routes-server (the backend API)
Now imagine you need to change the way a 'Booking' object looks — maybe you add a new field called cargo_weight. Here is what happens:
You update the Booking object in Repository 4 (server)
You update the code that displays bookings in Repository 1 (web)
You update the code in Repository 2 (agent)
You update the code in Repository 3 (operator)
You make four separate pull requests. Four separate code reviews. Four separate deployments.
One developer forgets to update Repository 2. The agent app breaks.
❌
The four-repo problem
Separate repositories create coordination nightmares. Teams get out of sync. The same code gets copied and pasted across repos and gets different bugs in each one. Shared code like the Booking type has to be updated in four places every time.
2.2  The Solution: One Repository, Many Apps
A monorepo puts ALL parts of the project in ONE GitHub repository. A single folder on your computer (and on GitHub) contains the traveller app, the agent app, the operator dashboard, the backend server, and shared code — all together.
  One GitHub Repository: naija-routes
  ├── apps/
  │   ├── web/          ← Traveller website
  │   ├── agent/        ← Park agent app
  │   └── operator/     ← Operator dashboard
  ├── server/           ← Backend API
  └── shared/           ← Code shared between all apps
Now when you change the Booking object, you change it ONCE in the shared/ folder. All four apps see the update automatically. One pull request. One code review. One deployment pipeline.
✅
The monorepo advantage
One change. One review. Everything stays in sync automatically. The shared/ folder contains types, validators, and constants that every app and the server can import. No more copy-pasting. No more four-repo chaos.
2.3  The Tool That Manages Our Monorepo: Turborepo
A monorepo with four apps and a server would be slow if every time you made a change, you had to rebuild all five parts. Turborepo solves this. It is a tool from Vercel that:
Only rebuilds the part you changed — if you only edited the agent app, only the agent app rebuilds
Caches build results — if nothing changed in the server, it reuses the last build
Runs tasks in parallel — it can build and test all apps at the same time
Understands dependencies — it knows the server must build before the apps, and handles the order automatically
You do not need to understand Turborepo deeply on day one. The configuration is already set up. You just run commands like npm run dev and Turborepo handles the rest.

The golden rule of file placement
If a file is only used by ONE app, it lives inside that app's folder. If a file is used by TWO OR MORE apps or by the server, it lives in the shared/ folder.

Section 4: How One Feature Works End to End
To make the architecture concrete, let us trace a single user action through the entire system — from tapping a button to seeing the result. We will use route search as the example because it touches the most parts.
The Journey: Passenger Searches for Lagos to Abuja Buses
STEP 1 — USER TAPS BUTTON
─────────────────────────
Passenger fills in: From: Lagos  To: Abuja  Date: 25-Dec
Taps the green 'Search Buses' button
File: apps/web/js/modules/search.js
 
STEP 2 — FRONTEND SENDS REQUEST
────────────────────────────────
search.js calls the API using the fetch wrapper:
  api.get('/api/v1/routes/search?origin=Lagos&destination=Abuja&date=2025-12-25')
File: apps/web/js/core/api.js
 
STEP 3 — REQUEST ARRIVES AT SERVER
────────────────────────────────────
Express receives the request.
Middleware runs first (in this order):
  1. cors.js      → checks the request is from an allowed website
  2. rateLimit.js → checks this user has not made too many requests
  3. validate.js  → checks origin, destination, date are present and valid
File: server/src/middleware/
 
STEP 4 — ROUTE HANDLER RUNS
────────────────────────────
search.routes.js receives the validated request.
It calls searchService.findRoutes(origin, destination, date)
File: server/src/routes/search.routes.js
 
STEP 5 — SERVICE QUERIES THE DATABASE AND CACHE
─────────────────────────────────────────────────
search.service.js first checks Redis cache:
  'Have we searched Lagos-Abuja-25Dec in the last 5 minutes?'
  YES → Return cached results instantly (no database hit)
  NO  → Query Typesense search engine for matching routes
        Query PostgreSQL for real-time seat availability
        Store result in Redis for 5 minutes
File: server/src/services/search.service.js
 
STEP 6 — SERVER RESPONDS
─────────────────────────
Server sends back a JSON array of matching operators:
  [ { operator: "ABC Transport", price: 9500, seats: 12, departure: "06:00" }, ... ]
 
STEP 7 — FRONTEND DISPLAYS RESULTS
────────────────────────────────────
search.js receives the JSON and builds result cards in HTML.
The user sees a list of operators with prices and times.
File: apps/web/js/modules/search.js
💡
The pattern for every feature
Every feature follows this same pattern: (1) User does something on the frontend. (2) Frontend calls the API. (3) Middleware validates the request. (4) Route handler calls a service. (5) Service talks to database/cache. (6) Server responds with data. (7) Frontend shows the data. Learn this pattern once and you understand every feature.
Section 5: Setting Up the Monorepo on GitHub — Step by Step
This section walks you through creating the Naija Routes monorepo on GitHub from scratch. Follow every step in order. Every command shown here is run in your computer's terminal (Command Prompt on Windows, Terminal on Mac/Linux).
⚠️
Before you start
You need these installed on your computer: Git, Node.js version 20 or later, and pnpm. If you do not have them, install them first. Instructions are at nodejs.org and pnpm.io.
Phase A: Create the GitHub Repository
A1
Create a GitHub account (if you don't have one)
Go to github.com and sign up with your email address. Choose a username — this will be visible publicly.
A2
Create the repository on GitHub
Click the green 'New' button on github.com. Fill in: Repository name: naija-routes. Description: Nigeria's road transport and logistics aggregator. Set it to Private (not Public — this is a business project). Do NOT tick 'Add a README file' — we will create our own. Click 'Create repository'.
A3
Copy the repository URL
After creating the repository, GitHub shows you a page with setup instructions. Copy the HTTPS URL — it looks like: https://github.com/YOUR-USERNAME/naija-routes.git
Phase B: Set Up the Project on Your Computer
B1
Open your terminal and create the project folder
Run these commands one at a time. Press Enter after each line.
# Create the project folder
mkdir naija-routes
 
# Go into the folder
cd naija-routes
 
# Tell Git this folder is a Git repository
git init
 
# Connect this folder to your GitHub repository
# (replace YOUR-USERNAME with your actual GitHub username)
git remote add origin https://github.com/YOUR-USERNAME/naija-routes.git
B2
Create the root package.json file
This file defines the project as a pnpm workspace (a monorepo). Create a file called package.json in the naija-routes folder with the following content:
{
  "name": "naija-routes",
  "version": "1.0.0",
  "private": true,
  "packageManager": "pnpm@9.0.0",
  "scripts": {
    "dev":   "turbo run dev",
    "build": "turbo run build",
    "test":  "turbo run test",
    "lint":  "turbo run lint"
  },
  "devDependencies": {
    "turbo": "^2.0.0"
  }
}
B3
Create the pnpm-workspace.yaml file
This tells pnpm which folders contain individual apps/packages. Create pnpm-workspace.yaml in the root folder:
packages:
  - 'apps/*'
  - 'server'
  - 'shared'
B4
Create the turbo.json file
This tells Turborepo how to run tasks across all packages. Create turbo.json in the root folder:
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "test": {
      "dependsOn": ["^build"]
    },
    "lint": {}
  }
}
B5
Create the .gitignore file
This tells Git which files to never upload to GitHub. VERY IMPORTANT — .env files contain secret API keys and must never go to GitHub. Create .gitignore in the root folder:
# Dependencies (installed locally, not stored in Git)
node_modules/
 
# Environment variables (SECRET — NEVER commit this)
.env
.env.local
.env.production
 
# Build output
dist/
.turbo/
 
# OS files
.DS_Store
Thumbs.db
 
# Editor files
.vscode/
.idea/
B6
Create the .env.example file
This is the TEMPLATE for environment variables. It lists every secret the project needs, but with fake placeholder values. Real values go in .env (which is never committed). Create .env.example in the root folder:

Phase E: Set Up Branch Protection (Very Important)
Branch protection rules prevent anyone from pushing code directly to the main branch without a review. This is essential for a team — without it, one mistake can break the entire project for everyone.
E1
Set up branch protection rules on GitHub
Go to your repository on GitHub and follow these steps:
Go to your repository: github.com/YOUR-USERNAME/naija-routes
Click Settings (the gear icon in the top navigation)
Click Branches in the left sidebar
Click Add branch protection rule
In 'Branch name pattern', type: main
Tick: Require a pull request before merging
Tick: Require at least 1 approval (so 1 teammate must review your code before it merges)
Tick: Require status checks to pass before merging (this ensures tests must pass)
Tick: Do not allow bypassing the above settings
Click Create at the bottom
Phase F: Set Up GitHub Actions (Automated Testing)
GitHub Actions automatically runs your tests every time anyone pushes code. This catches bugs before they reach the main branch. Create this file exactly as shown:
F1
Create the CI workflow file
Create the file at this exact path: .github/workflows/ci.yml
name: CI — Test and Lint
 
# Run this workflow every time code is pushed or a pull request is opened
on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]
 
jobs:
  test:
    name: Test and Lint
    runs-on: ubuntu-latest
 
    steps:
      # Step 1: Download the code
      - name: Checkout code
        uses: actions/checkout@v4
 
      # Step 2: Install Node.js
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
 
      # Step 3: Install pnpm
      - name: Install pnpm
        uses: pnpm/action-setup@v3
        with:
          version: 9
 
      # Step 4: Install all dependencies
      - name: Install dependencies
        run: pnpm install
 
      # Step 5: Run linting (checks code style)
      - name: Lint
        run: pnpm lint
 
      # Step 6: Run tests
      - name: Test
        run: pnpm test
F2
Commit and push the workflow file
After creating the workflow file, commit and push it to GitHub.
git add .github/workflows/ci.yml
git commit -m "ci: add GitHub Actions CI workflow"
git push
 
# Go to github.com/YOUR-USERNAME/naija-routes/actions
# You should see the workflow listed there
Phase G: The Team Workflow — How Everyone Works Together
Now that the repository is set up, here is the workflow every developer must follow. Never push directly to main. Always work on a feature branch.
THE DAILY WORKFLOW FOR EVERY DEVELOPER
═══════════════════════════════════════
 
1. PULL LATEST CODE before starting any work
   git checkout main
   git pull origin main
 
2. CREATE A BRANCH for your feature
   # Branch naming: type/short-description
   # Types: feat (new feature), fix (bug fix), chore (maintenance)
   git checkout -b feat/add-route-search
 
3. WRITE YOUR CODE in the correct folder
   # Example: if adding the route search API endpoint,
   # you edit: server/src/routes/search.routes.js
   #       and: server/src/services/search.service.js
 
4. COMMIT YOUR CHANGES with a clear message
   git add .
   git commit -m "feat: add route search endpoint with Typesense integration"
 
5. PUSH YOUR BRANCH to GitHub
   git push origin feat/add-route-search
 
6. OPEN A PULL REQUEST on GitHub
   Go to github.com/YOUR-USERNAME/naija-routes
   Click 'Compare & pull request'
   Write a description of what you changed and why
   Request a review from a teammate
 
7. TEAMMATE REVIEWS AND APPROVES
   They leave comments if anything needs fixing
   You fix and push more commits to the same branch
   When they approve, you can merge
 
8. MERGE TO MAIN
   Click 'Merge pull request' on GitHub
   Delete the branch after merging (keep things clean)
 
9. EVERYONE PULLS THE UPDATE
   All teammates run: git pull origin main
   to get your merged changes
